# AI Agent 完整架构 — 从代码到模型

## 概述

本文档详细描述了 AI Agent（如 Claude Code CLI）从用户代码到实际模型推理的完整调用链路，包括：
- 协议格式（Anthropic Messages API vs OpenAI Chat Completions）
- 各层次的角色和职责
- Gateway 的作用
- 模型提供商和服务端的关系

---

## 一、协议格式详解

### 1.1 什么是协议？

**协议是 LLM API 的消息格式标准，规定了请求和响应的数据结构。**

不同 LLM 提供商使用不同的协议：

| 提供商 | API 协议 | 文档地址 |
|--------|---------|---------|
| Anthropic | Messages API | https://docs.anthropic.com/en/api/messages |
| OpenAI | Chat Completions API | https://platform.openai.com/docs/api-reference/chat |
| Google Gemini | Gemini API | https://ai.google.dev/api/generate-content |

### 1.2 Anthropic Messages API 格式

#### 请求格式

```python
# User 消息
{
    "role": "user",       # 必填，"user" 或 "assistant"
    "content": "文本" | [ContentBlock...]  # 必填，字符串或 block 数组
}

# 工具定义
{
    "name": "bash",
    "description": "Run a shell command.",
    "input_schema": {       # ← 用 input_schema
        "type": "object",
        "properties": {"command": {"type": "string"}},
        "required": ["command"]
    }
}
```

#### 响应格式

```python
# Assistant 消息（LLM 返回）
{
    "id": "msg_xxx",
    "role": "assistant",
    "content": [
        {"type": "text", "text": "这是文本内容"},
        {"type": "thinking", "thinking": "思考过程"},
        {"type": "tool_use", "id": "toolu_xxx", "name": "bash", "input": {"command": "ls"}}
    ],
    "stop_reason": "tool_use" | "end_turn"
}

# Tool Result（用户回传）
{
    "role": "user",
    "content": [
        {"type": "tool_result", "tool_use_id": "toolu_xxx", "content": "执行结果"}
    ]
}
```

### 1.3 OpenAI Chat Completions API 格式

#### 请求格式

```python
# User 消息
{
    "role": "user",
    "content": "文本"
}

# 工具定义
{
    "type": "function",      # ← 必须指定 type
    "function": {
        "name": "bash",
        "description": "Run a shell command.",
        "parameters": {       # ← 用 parameters，不是 input_schema
            "type": "object",
            "properties": {"command": {"type": "string"}},
            "required": ["command"]
        }
    }
}
```

#### 响应格式

```python
# Assistant 消息（LLM 返回）
{
    "id": "chatcmpl_xxx",
    "choices": [{
        "message": {
            "role": "assistant",
            "content": None,  # ← 调用工具时为 None
            "tool_calls": [{  # ← 用 tool_calls 数组
                "id": "call_xxx",
                "type": "function",
                "function": {
                    "name": "bash",
                    "arguments": '{"command": "ls"}'  # ← JSON 字符串
                }
            }]
        }
    }]
}

# Tool Result（用户回传）
{
    "role": "tool",           # ← 用 "tool" 角色
    "tool_call_id": "call_xxx",  # ← 用 tool_call_id
    "content": "执行结果"
}
```

### 1.4 关键差异对比

| 方面 | Anthropic Messages | OpenAI Chat Completions |
|------|--------------------|------------------------|
| **tools 定义字段** | `input_schema` | `parameters` |
| **tools 类型声明** | 不需要 type | 必须有 `type: "function"` |
| **tool 调用位置** | `content` 数组内 | `tool_calls` 数组内 |
| **tool 调用时 content** | 有值（包含 tool_use） | `None` |
| **arguments 格式** | `input`（JSON **对象**） | `arguments`（JSON **字符串**） |
| **tool result role** | `"user"` | `"tool"` |
| **ID 字段名** | `tool_use_id` | `tool_call_id` |

### 1.5 字段详解

#### role 字段

| 值 | 来源 | 说明 |
|---|------|------|
| `"user"` | 程序指定 | 用户消息，包括 tool_result |
| `"assistant"` | API 自动填充 | LLM 返回的消息 |
| `"tool"` (OpenAI) | 程序指定 | OpenAI 专用的 tool result 角色 |

#### content 字段

```python
# Anthropic 格式
# content 可以是两种形式：

# 1. 简单字符串（只有文本）
content = "这是一段文本"

# 2. ContentBlock 数组（多个 block）
content = [
    {"type": "text", "text": "..."},
    {"type": "thinking", "thinking": "..."},
    {"type": "tool_use", "id": "...", "name": "...", "input": {...}}
]
```

#### type 字段（ContentBlock 类型）

| type | 来源 | 说明 |
|------|------|------|
| `"text"` | LLM 生成 | 文本内容 |
| `"thinking"` | LLM 生成 | 思考过程（Claude 3.7+） |
| `"tool_use"` | LLM 生成 | 工具调用请求 |
| `"tool_result"` | 程序指定 | 工具执行结果 |

#### tool_use_id 字段

**重要：tool_use_id 是 API 自动生成的，用于关联 tool_use 和 tool_result！**

```python
# LLM 返回 tool_use block
{
    "type": "tool_use",
    "id": "toolu_eca74706bb414578acaa8739",  # ← API 生成的唯一 ID
    "name": "write_file",
    "input": {"path": "test.py", "content": "..."}
}

# 程序返回 tool_result 时，必须用同一个 ID
{
    "type": "tool_result",
    "tool_use_id": "toolu_eca74706bb414578acaa8739",  # ← 关联到上面的 tool_use
    "content": "Wrote 15 bytes..."
}
```

**为什么需要 ID？**

```python
# 同一轮可能调用多个同名工具
[
    {"type": "tool_use", "id": "toolu_111", "name": "read_file", "input": {"path": "a.py"}},
    {"type": "tool_use", "id": "toolu_222", "name": "read_file", "input": {"path": "b.py"}}
]

# 返回结果时用 ID 区分
[
    {"type": "tool_result", "tool_use_id": "toolu_111", "content": "内容A"},
    {"type": "tool_result", "tool_use_id": "toolu_222", "content": "内容B"}
]
```

#### name 字段

**name 来自 TOOLS 定义**

```python
# 程序定义 TOOLS
TOOLS = [
    {"name": "bash", ...},       # ← LLM 返回时用这个 name
    {"name": "read_file", ...},  # ← LLM 返回时用这个 name
]

# LLM 返回
{"type": "tool_use", "name": "read_file", ...}  # ← 匹配 TOOLS 中的定义
```

### 1.6 市场占有率

| 协议 | 市场情况 |
|------|---------|
| **OpenAI Chat Completions** | **行业标准**，绝大多数服务兼容 |
| Anthropic Messages | Anthropic 专用，但越来越多服务提供兼容端点 |

**为什么 OpenAI 格式更流行？**

1. **历史原因**：OpenAI 最早推出 API（2020 年），定义了行业标准
2. **兼容性**：几乎所有第三方服务都支持 OpenAI 格式
3. **简单易懂**：格式相对直观

**兼容性列表：**

| 服务 | 支持 OpenAI 格式 | 支持 Anthropic 格式 |
|------|-----------------|--------------------|
| OpenAI | ✅ 原生 | ❌ |
| Anthropic Claude | ❌ | ✅ 原生 |
| DeepSeek | ✅ 原生 | ✅（通过 /anthropic 端点） |
| Moonshot (Kimi) | ✅ 原生 | ✅（通过 /anthropic 端点） |
| GLM (智谱) | ✅ 原生 | ✅（通过 /anthropic 端点） |
| Minimax | ✅ 原生 | ✅（通过 /anthropic 端点） |

### 2.1 架构概览

```
┌─────────────────────────────────────────────────────────────┐
│ 第 1 层：用户程序                                             │
│                                                              │
│ Claude Code CLI / Python 代码                                │
│     ├─ 使用 Anthropic SDK                                    │
│     ├─ 发送 Anthropic Messages 格式                          │
│     └─ 期望收到 Anthropic Messages 格式                      │
│                                                              │
│ 配置：                                                       │
│     ANTHROPIC_BASE_URL=https://devops-llmgateway.sto.cn     │
│     MODEL_ID=dashscope/glm-5.1                              │
│     ANTHROPIC_API_KEY=xxx                                    │
└─────────────────────────────────────────────────────────────┘
                    ↓ HTTPS (Anthropic Messages 协议)
                    ↓
┌─────────────────────────────────────────────────────────────┐
│ 第 2 层：企业 Gateway (适配层)                                │
│                                                              │
│ devops-llmgateway.sto.cn                                     │

│     ├─ 安全审计                                               │
│     └─ API Key 集中管理                                       │
└─────────────────────────────────────────────────────────────┘
                    ↓ HTTPS (目标协议，如 OpenAI)
                    ↓
┌─────────────────────────────────────────────────────────────┐
│ 第 3 层：模型服务端（托管平台）                                │
│                                                              │
│ 阿里云 DashScope / 智谱服务端 / Anthropic 服务端              │
│     ├─ 会员管理（账户系统）                                   │
│     ├─ 计费系统（按 token 计费）                              │
│     ├─ 流量控制（Rate Limiting）                              │
│     ├─ 安全检查（内容过滤）                                   │
│     ├─ 负载均衡（分发到多个模型实例）                          │
│     └─ 模型调度                                               │
└─────────────────────────────────────────────────────────────┘
                    ↓ 内部协议
                    ↓
┌─────────────────────────────────────────────────────────────┐
│ 第 4 层：模型集群                                             │
│                                                              │
│ GLM-5.1 / Claude Sonnet / GPT-4                              │
│     ├─ 模型推理（GPU 计算）                                   │

│     └─ 不直接暴露给外部                                       │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 各层次详解

#### 第 1 层：用户程序

**特点：**
- 使用固定的 SDK（如 Anthropic SDK）
- 发送固定的协议格式（Anthropic Messages）
- 配置决定了请求发往哪里

**关键配置：**

```bash
# 环境变量配置
ANTHROPIC_API_KEY=xxx           # API Key（认证）
ANTHROPIC_BASE_URL=xxx          # 请求地址（决定发往哪个 Gateway）
MODEL_ID=dashscope/glm-5.1      # 模型标识（决定使用哪个模型）
```

#### 第 2 层：企业 Gateway（适配层）

**角色定位：**
- **协议转换器**：在不同协议格式之间转换
- **企业统一入口**：集中管理所有模型访问
- **适配器模式**：让固定协议的客户端可以访问多种模型

**核心功能：**

| 功能 | 说明 |
|------|------|
| **协议转换** | Anthropic Messages ↔ OpenAI Chat Completions |
| **模型路由** | 根据 MODEL_ID 前缀分发到不同提供商 |
| **会员管理** | 企业内部用户认证 |
| **成本统计** | 记录每个团队的 API 使用量 |
| **流量控制** | 防止某个团队过度使用 |
| **安全审计** | 记录所有请求内容 |
| **API Key 管理** | 集中保管各提供商的 API Key，不暴露给开发者 |

**协议转换示例：**

```python
# Gateway 收到的请求（Anthropic 格式）
{
    "model": "dashscope/glm-5.1",
    "messages": [{"role": "user", "content": "创建文件"}],
    "tools": [{"name": "bash", "input_schema": {...}}]
}

# Gateway 解析 MODEL_ID，路由到 DashScope
# 转换成 OpenAI 格式
{
    "model": "glm-5.1",
    "messages": [{"role": "user", "content": "创建文件"}],
    "tools": [{"type": "function", "function": {"name": "bash", "parameters": {...}}}]
}

# 发送给 DashScope
# 收到 DashScope 响应（OpenAI 格式）
{
    "choices": [{
        "message": {
            "tool_calls": [{"function": {"name": "bash", "arguments": "..."}}]
        }
    }]
}

# Gateway 转换成 Anthropic 格式
{
    "content": [{"type": "tool_use", "name": "bash", "input": {...}}],
    "stop_reason": "tool_use"
}

# 返回给用户程序
```

#### 第 3 层：模型服务端（托管平台）

**角色定位：**
- 类似于 Anthropic 服务端
- 负责会员、计费、流量、安全等
- 管理模型集群

**典型服务端：**

| 服务端 | 提供商 | 支持的模型 |
|--------|-------|-----------|
| Anthropic 服务端 | Anthropic | Claude 系列 |
| 阿里云 DashScope | 阿里巴巴 | 通义千问、GLM、Baichuan、Llama 等 |
| 智谱服务端 | 智谱AI | GLM 系列 |
| DeepSeek 服务端 | DeepSeek | DeepSeek 系列 |
| OpenAI 服务端 | OpenAI | GPT 系列 |

**核心功能：**

| 功能 | 说明 |
|------|------|
| **会员管理** | 用户账户、API Key 验证 |
| **计费系统** | 按 token 计费、账户余额 |
| **流量控制** | Rate Limiting（每秒请求数限制） |
| **安全检查** | 内容过滤、敏感词检测 |
| **负载均衡** | 分发到多个模型实例 |
| **模型调度** | 选择合适的模型实例 |

#### 第 4 层：模型集群

**特点：**
- 实际执行推理的 GPU 服务器集群
- 不直接暴露给外部，由服务端管理
- 多地域、多实例部署

**部署位置：**

| 模型 | 部署位置 |
|------|---------|
| Claude | Anthropic 数据中心 |
| GLM（通过智谱） | 智谱数据中心 |
| GLM（通过 DashScope） | 阿里云数据中心 |
| GPT | OpenAI 数据中心（微软 Azure） |

## 三、Gateway 的作用详解

### 3.1 为什么需要 Gateway？

#### 没有 Gateway 的问题

```
用户程序 → 直接请求各模型服务端

问题：
    ├─ 代码必须适配多种 SDK（Anthropic、OpenAI、智谱...）
    ├─ Claude Code CLI 只支持 Anthropic 格式，无法使用其他模型
    ├─ API Key 直接暴露给开发者（安全风险）
    ├─ 没有企业内部的成本统计、流量控制
    ├─ 每个开发者要自己管理多个 API Key
    └─ 无法统一管理、审计
```

#### 有 Gateway 的好处

```
用户程序 → Gateway → 各种模型服务端

好处：
    ├─ 统一用一种 SDK（如 Anthropic SDK）
    ├─ Claude Code CLI 可以使用任何模型
    ├─ 企业内部统一管理（成本、流量、安全）
    ├─ API Key 集中管理（不暴露给开发者）
    ├─ 可以随时切换模型（不改代码）
    ├─ 可以同时使用多个模型提供商
    └─ 统一审计、合规管理
```

### 3.2 Gateway 与 Anthropic 服务端的对比

| 角色 | Anthropic 服务端 | 企业 Gateway |
|------|-----------------|-------------|
| **主要职责** | 为 Claude 模型提供服务 | 协议转换 + 企业内部管理 |
| **会员管理** | Anthropic 用户账户 | 企业内部用户 |
| **计费** | Anthropic 计费系统 | 企业内部成本统计 |
| **流量控制** | Rate Limiting | 企业内部流量管理 |
| **安全检查** | 内容过滤 | 企业内部审计 |
| **协议转换** | ❌（原生 Anthropic） | ✅ 核心功能 |
| **模型归属** | 自己的模型 | 调用第三方模型 |

**关键区别：Anthropic 服务端是模型提供商，Gateway 是企业内部的适配层。**

## 四、模型开发商与托管平台

### 4.1 重要概念：模型开发商 ≠ 托管平台

```
模型开发商：
    ├─ 开发模型的团队（如智谱AI）
    ├─ 拥有模型的技术和知识产权
    └─ 可以自己提供服务端

托管平台：
    ├─ 提供 GPU 计算资源和 API 服务
    ├─ 托管多种模型（包括第三方模型）
    ├─ 统一计费、会员管理
    └─ 向模型开发商支付授权分成
```

### 4.2 典型案例：GLM 模型

#### 模型开发商：智谱AI

```
智谱AI（北京智谱华章科技有限公司）

公司信息：
    ├─ 成立于 2019 年
    ├─ 总部：北京
    ├─ 清华大学技术团队
    └─ 独立的大模型公司

模型：
    ├─ GLM-4
    ├─ GLM-5
    ├─ GLM-5.1
    └─ ChatGLM 系列

自己的服务端：
    ├─ 国内：https://open.bigmodel.cn
    ├─ 国际：https://z.ai
    ├─ 自己部署、自己管理、自己计费
```

#### 托管平台：阿里云 DashScope

```
阿里云 DashScope

公司信息：
    ├─ 阿里巴巴旗下
    ├─ 模型服务平台
    └─ 类似于 Azure OpenAI 的模式

服务端：
    ├─ https://dashscope.aliyuncs.com
    └─ 部署在阿里云数据中心

托管的模型（多种来源）：
    ├─ 通义千问（阿里自己的模型）
    ├─ GLM（智谱授权托管）
    ├─ Baichuan（百川授权托管）
    ├─ Llama（Meta 开源）
    └─ 其他模型

商业模式：
    ├─ 模型托管平台
    ├─ 统一 API 接口
    ├─ 阿里云账户计费
    └─ 向模型开发商支付分成
```

### 4.3 两种使用 GLM 的方式

#### 方式 1：直接使用智谱服务端

```
用户程序 → 智谱服务端 (open.bigmodel.cn) → GLM 模型

特点：
    ├─ 智谱账户管理
    ├─ 智谱计费
    ├─ 智谱流量控制
    └─ 直连模型开发商
```

#### 方式 2：通过阿里云 DashScope 使用 GLM

```
用户程序 → 阿里云 DashScope → GLM 模型（智谱授权）

特点：
    ├─ 阿里云账户管理
    ├─ 阿里云计费
    ├─ 阿里云流量控制
    ├─ 模型在阿里云 GPU 上运行
    └─ 智谱获得授权分成
```

### 4.4 MODEL_ID 的含义

```bash
MODEL_ID=dashscope/glm-5.1
```

| 部分 | 含义 |
|------|------|
| `dashscope/` | 通过阿里云 DashScope 平台访问 |
| `glm-5.1` | 使用 GLM-5.1 模型（智谱开发） |

**其他示例：**

```bash
MODEL_ID=openai/gpt-4           # 通过 OpenAI 访问 GPT-4
MODEL_ID=anthropic/claude-sonnet  # 通过 Anthropic 访问 Claude
MODEL_ID=zhipu/glm-5.1          # 直接通过智谱访问 GLM
MODEL_ID=deepseek/deepseek-chat # 通过 DeepSeek 访问
```

### 4.5 类似的托管模式

| 托管平台 | 托管的模型（第三方） |
|---------|--------------------|
| Azure OpenAI | GPT 系列（OpenAI） |
| 阿里云 DashScope | GLM（智谱）、Baichuan（百川）、Llama（Meta） |
| AWS Bedrock | Claude（Anthropic）、Llama（Meta） |
| Google Vertex AI | Claude（Anthropic）、Gemini（Google） |

## 五、完整请求流程示例

### 5.1 用户配置

```bash
# .env 配置
ANTHROPIC_API_KEY=sk-xxx
ANTHROPIC_BASE_URL=https://devops-llmgateway.sto.cn
MODEL_ID=dashscope/glm-5.1
```

### 5.2 完整流程

```
┌─────────────────────────────────────────────────────────────┐
│ 步骤 1：用户程序发送请求                                       │
│                                                              │
│ Claude Code CLI                                              │
│     ↓ 使用 Anthropic SDK                                     │
│                                                              │
│ POST https://devops-llmgateway.sto.cn/v1/messages           │
│ Headers:                                                     │
│     Authorization: Bearer sk-xxx                            │
│     Content-Type: application/json                          │
│ Body (Anthropic Messages 格式):                              │
│ {                                                            │
│   "model": "dashscope/glm-5.1",                              │
│   "messages": [{"role": "user", "content": "创建文件"}],      │
│   "tools": [{"name": "bash", "input_schema": {...}}]         │
│ }                                                            │
└─────────────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────────────┐
```

## 六、CC-Switch 的作用

### 6.1 CC-Switch 是什么？

[CC-Switch](https://github.com/dyskette/cc-switch) 是一个 CLI 工具，用于配置 Claude Code CLI 的模型提供商。

### 6.2 CC-Switch 配置的对象

**关键理解：CC-Switch 配置的是 Gateway 与目标模型通信的协议，不是 Claude Code CLI 的协议！**

```
┌─────────────────────────────────────────────────────────────┐
│ 协议配置层次                                                  │
│                                                              │
│ 第一层：Claude Code CLI → Gateway                             │
│   ├─ 固定使用 Anthropic Messages 协议                        │
│   ├─ 这是 Claude Code 内部决定的，不可更改                   │
│   └─ 因为 Claude Code 是为 Anthropic Claude 设计的           │
│                                                              │
│ 第二层：Gateway → 目标模型                                    │
│   ├─ 由 CC-Switch 配置决定                                   │
│   ├─ 可以是 OpenAI、Anthropic、自定义等                      │
│   └─ Gateway 根据配置转换协议                                │
└─────────────────────────────────────────────────────────────┘
```

### 6.3 CC-Switch 配置示例

```
CC-Switch 配置界面：
    ├─ Provider: DashScope
    ├─ Model: glm-5.1
    ├─ API Format: OpenAI Chat Completions  ← 配置 Gateway 与 DashScope 的协议
    ├─ API Key: xxx
    └─ Base URL: https://dashscope.aliyuncs.com

实际效果：
    Claude Code CLI 发送 Anthropic 格式 → Gateway → 转成 OpenAI 格式 → DashScope
```

## 七、架构对比总结

### 7.1 直接使用 Anthropic Claude

```
Claude Code CLI → Anthropic 服务端 → Claude 模型

特点：
    ├─ 协议：Anthropic Messages（原生）
    ├─ 账户：Anthropic 账户
    ├─ 计费：Anthropic 计费
    ├─ 模型：Anthropic 自己的
```

### 7.2 通过 Gateway 使用 GLM

```
Claude Code CLI → Gateway → DashScope → GLM 模型

特点：
    ├─ 协议：Anthropic Messages（CLI）+ OpenAI（Gateway↔DashScope）
    ├─ 账户：企业账户（Gateway）+ 阿里云账户（DashScope）
    ├─ 计费：企业统计（Gateway）+ 阿里云计费（DashScope）
    ├─ 模型：智谱开发的 GLM，阿里云托管
```

### 7.3 层次对比表

| 层次 | Anthropic 方案 | Gateway 方案 |
|------|---------------|-------------|
| **第 1 层：程序** | Claude Code CLI | Claude Code CLI |
| **第 2 层：中间层** | - | **企业 Gateway**（协议转换） |
| **第 3 层：服务端** | Anthropic 服务端 | 阿里云 DashScope 服务端 |
| **第 4 层：模型提供商** | Anthropic（同一公司） | **智谱AI**（独立公司） |
| **第 5 层：模型** | Claude 模型 | GLM 模型（智谱授权托管） |
| **部署运维** | Anthropic | 阿里云 |

## 八、核心概念总结

### 8.1 关键角色

| 角色 | 说明 |
|------|------|
| **用户程序** | Claude Code CLI，固定使用 Anthropic Messages 协议 |
| **Gateway** | 企业内部的适配层，协议转换 + 企业管理 |
| **模型服务端** | 托管平台，会员管理、计费、流量控制（如 DashScope） |
| **模型开发商** | 模型的开发者和知识产权拥有者（如智谱AI） |
| **模型集群** | 实际推理的 GPU 服务器，不直接暴露 |

### 8.2 关键关系

| 关系 | 说明 |
|------|------|
| **Gateway vs 服务端** | Gateway 是适配层，服务端是模型托管平台 |
| **开发商 vs 托管平台** | 开发商提供模型，托管平台提供服务 |
| **协议层次** | 第一层（CLI↔Gateway）固定，第二层（Gateway↔服务端）可配置 |

### 8.3 关键概念

| 概念 | 说明 |
|------|------|
| **协议转换** | Gateway 的核心功能，在不同 API 格式之间转换 |
| **MODEL_ID** | 包含托管平台前缀和模型名（如 dashscope/glm-5.1） |
| **tool_use_id** | API 自动生成，用于关联 tool_use 和 tool_result |
| **托管模式** | 托管平台提供 GPU 资源和 API 服务，向开发商支付分成 |

## 九、类比理解

### 9.1 网购类比

```
你用支付宝买星巴克咖啡：

你（用户）= Claude Code CLI
支付宝（托管平台）= 阿里云 DashScope
星巴克（品牌/开发商）= 智谱AI（GLM 模型）
咖啡（实际产品）= GLM 模型推理

支付给支付宝 = 付给阿里云
星巴克获得分成 = 智谱获得分成
```

### 9.2 Azure OpenAI 类比

```
Azure OpenAI 模式：
    微软 Azure（托管平台）托管 OpenAI 的 GPT 模型
    → 用户通过 Azure 账户访问
    → OpenAI 获得授权分成

阿里云 DashScope 模式：
    阿里云（托管平台）托管智谱的 GLM 模型
    → 用户通过阿里云账户访问
    → 智谱获得授权分成
```
